What is CI/CD?

Continuous integration

Continuous integration (CI) is a software practice that requires frequently committing code to a repository. Committing code more often can detect errors sooner and reduce the amount of code a developer needs to debug when finding the source of an error. This is great for developers, so they can spend more time writing code and less time debugging errors or resolving merge conflicts.
When you commit code to your repository, you can continuously build and test that code to make sure that the commit doesn't introduce any errors.
Building and testing code requires a server. You can build and test locally before pushing code, or you can use a CI server that checks for new code commits in a repository.

Continuous deployment

Continuous deployment (CD) builds on continuous integration. When new code is committed and passes the CI tests, the code is automatically deployed to production. You can create custom CD workflows to automatically deploy your code to any cloud, self-hosted service, or platform from your repository, which saves developers time by automating the deployment process and deploying tested, stable code changes more quickly to your customers.

Automated Builds

Build automation is the process of automating the creation of a software build and the associated processes including: compiling computer source code into binary code, packaging binary code, and running automated tests.
There are many build automation softwares, like GNU make, make, CMake, GNU Build System...
For a complete list click here.

GitHub Actions

GitHub Actions help you automate your software development workflows in the same place you store code. You can write individual tasks, called actions, and combine them to create a custom workflow.
Having your workflows and your automations bound to GitHub has many benefits, as for example, when you push corrupted code to the repository, the automatic build software can build it and test it and, upon seeing it's corrupted, dtop the push and don't let it get into the repository.
GitHub Marketplace is where you can find, share, and use actions built by the GitHub community.
You can also customize your project with open source actions shared in public repositories and use actions built by the GitHub Actions organization.
The only downside is that Github and the community don't like Windows or C/C++ much, so you won't find as many actions already developed for you if you are using them instead of others.

What is a workflow?

Workflows are custom automated processes that you can set up in your repository to build, test, package, release, or deploy any code project on GitHub.
Workflows are made up of one or more jobs and can be scheduled or activated by events, like a push, a pull request or a fork.
They are .yml or .yaml files

GitHub Actions starter workflow

The starter workflow GitHub Actions gives you if you use C/C++ uses Make for the automated build. That means you code will now require a Makefile, which isn't quite difficult if your project is really simple, but gets pretty hard quite fast as the project gets bigger.
For being a starter workflow it can get quite hard if you don't know what you are doing, and there isn't any specific documentation available on GitHub.
But, ultimately, it's a simple workflow easy to implement and easy to use if you know what you are doing.

Create our own workflow

The upside of vreating your own workflow is that you can use whichever automation build you like, as long as it works with the language you are using on your code.
All of them require their own specific files, with their own specific syntax, just as Make needed it's Makefile, to be able to build the code.
It still isn't a very good option for beginners. Here there's a workflow syntax guide for GitHub Actions in case you want to try.

CMake

CMake is a build automation software that generates files for various build tools, such as Make, Ninja, Xcode and Visual Studio.
It's a really well documented software, so it's easy to learn, via the official documentation or via online forums wher people help eachother with whichever doubt they may have.
The only downside is that it's conceived to be creating the porjects on the same CMake, so it can get a little tricky when trying to use a project created from Visual Studio, for example, but you can get around it pretty easily.

More uses for GitHub Actions

You can get to do practically everything with GitHub Actions, even without knowing much about creating your own workflows, as you can use diverse actions already made by the community and combine them using diverse events to trigger them too.
But you have to be careful with this, because this combinations can make it all go wrong too.
